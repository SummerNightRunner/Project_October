# Архитектурные решения

## 2026-06-13 - Чистое базовое состояние вместо продолжения хаотичных результатов задач

Статус: принято.

Решение:

Сначала привести репозиторий к понятному базовому состоянию: данные, документация, `.gitignore`, FastAPI endpoint `GET /health` и текущий рекомендательный прототип.

Причина:

Предыдущие автоматические запуски задач оставили противоречивое состояние: часть документов отсутствовала, API-001 не подтверждалась в текущей рабочей копии, а данные были перенесены без зафиксированной политики.

## 2026-06-13 - Исходные CSV живут в `data/raw/`

Статус: принято.

Решение:

Считать `data/raw/` целевой директорией для исходных CSV. Файлы отслеживаются через Git LFS.

Последствия:

- Backend-код читает исходные данные из `data/raw/`.
- Старые отслеживаемые пути `data/*.csv` заменяются на `data/raw/*.csv`.
- Генерируемые файлы в `data/processed/` не коммитятся.

## 2026-06-13 - PostgreSQL как целевое хранилище

Статус: принято.

Решение:

Для пользовательской истории, рейтингов, API keys и состояния приложения целевым хранилищем будет PostgreSQL.

Последствия:

- CSV остается источником исходных данных о фильмах, но не хранилищем состояния приложения.
- SQLite можно использовать только как временный локальный эксперимент, если это явно согласовано.

## 2026-06-13 - Базовая схема пользовательской истории

Статус: принято.

Решение:

Для DB-001 принята базовая PostgreSQL-схема:

- `users` хранит аккаунты пользователей без проектирования паролей и внешней авторизации на этом этапе;
- `movie_catalog_entries` хранит минимальную проекцию локального каталога для ссылочной целостности;
- `user_movie_history` хранит актуальный факт истории пользователя по фильму;
- `user_movie_ratings` хранит нормализованную оценку пользователя по шкале `0..10`;
- `user_preferences` хранит ручные предпочтения с положительным или отрицательным весом;
- `api_clients` и `api_keys` разделяют сторонний проект и его ключи доступа;
- `user_events` хранит аудит действий и не является источником истины для истории или оценок.

Причина:

Рекомендательному MVP нужно отделить состояние приложения от локальных CSV-артефактов и подготовить хранение истории, оценок, ручных предпочтений и API-доступа без немедленного внедрения миграций.

Последствия:

- Пользовательские данные ссылаются на фильмы через `catalog_movie_id`, который равен строковому `id` из `data/processed/processed_metadata.csv`.
- API-ключи должны храниться только как `key_prefix` и `key_hash`; полный секрет в базе не хранится.
- Следующие backend-задачи должны добавить миграции, подключение PostgreSQL, синхронизацию `movie_catalog_entries` и API endpoints для истории.

## 2026-06-13 - DB-инфраструктура на SQLAlchemy, Alembic и psycopg

Статус: принято.

Решение:

Для backend DB-слоя использовать SQLAlchemy 2.x, Alembic и PostgreSQL-драйвер `psycopg` 3.x. Подключение к базе задается только через переменную окружения `PROJECT_OCTOBER_DATABASE_URL`.

Причина:

Этот стек является стандартным для FastAPI/Python backend, поддерживает явные миграции, PostgreSQL-first подход и не требует хранить секреты или локальные DSN в репозитории.

Последствия:

- FastAPI app не подключается к базе при импорте и продолжает работать без `PROJECT_OCTOBER_DATABASE_URL`.
- SQLAlchemy engine/session создаются лениво только когда DB-код явно запрошен.
- Alembic окружение готово для будущих миграций, но DB-002 не создает пользовательские таблицы из DB-001.
- Реальные модели, миграции пользовательской истории и синхронизация каталога остаются задачами DB-003 и DB-004.

## 2026-06-13 - Миграция пользовательской истории PostgreSQL-first

Статус: принято.

Решение:

DB-003 реализует схему DB-001 через SQLAlchemy ORM-модели и Alembic revision `db003_user_history_schema`. Миграция использует PostgreSQL-типы `uuid`, `citext`, `jsonb` и `text[]`, а также extensions `pgcrypto` и `citext`.

Причина:

Пользовательская история, оценки, предпочтения, API-доступ и события должны иметь явный миграционный контракт до реализации endpoints и синхронизации каталога.

Последствия:

- `Base.metadata` содержит таблицы DB-001 и используется Alembic для будущих проверок схемы.
- `movie_catalog_entries` создана как FK-цель, но ее заполнение из `processed_metadata.csv` остается DB-004.
- API endpoints пользовательской истории не реализуются в DB-003 и остаются API-004.
- SQLite-прогон миграции не является целевым из-за PostgreSQL-специфичных типов; безопасная проверка выполняется через импорт ревизии, `alembic history` и offline SQL generation.

## 2026-06-13 - Синхронизация минимального каталога без удаления строк

Статус: принято.

Решение:

DB-004 синхронизирует `movie_catalog_entries` из `processed_metadata.csv` отдельным сервисом и CLI entrypoint `python -m backend.app.db.sync_movie_catalog`. Синхронизация делает upsert по `catalog_movie_id`, заполняет только минимальные поля FK-проекции и не удаляет строки, которых больше нет в CSV.

Причина:

`movie_catalog_entries` нужна как стабильная FK-цель для пользовательской истории, но не должна становиться копией рекомендательного датасета или хранить модельные признаки. Удаление строк при смене локального CSV может нарушить ссылочную целостность будущей истории и оценок.

Последствия:

- Новые фильмы из CSV добавляются в `movie_catalog_entries`.
- Существующие фильмы обновляют `title_snapshot`, `release_date`, `source_catalog_version` и `updated_at`; `created_at` сохраняется.
- `source_catalog_version` остается `NULL`, если оператор явно не передал стабильную версию через CLI или сервисный параметр.
- Схема DB-003 не меняется, новая миграция не нужна.
- API endpoints пользовательской истории не реализуются в DB-004 и остаются задачей API-004.
